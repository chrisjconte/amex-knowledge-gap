Solution Overview

Design Principle: The platform detects (OOB). The KM team decides which gaps to act on (human-in-the-loop). The flow orchestrates the drafting, routing, and review cycle (custom). Humans approve every publish.

Detect (OOB)

Knowledge Demand Insights — an OOB ML capability in core Knowledge Management — clusters resolved incidents with no linked KB article. Results surface in the Knowledge Center "Top 5 potential gaps" tab automatically.

Draft (OOB Skill)

KM team selects a gap and clicks [Generate Article] in the Knowledge Center. The OOB Knowledge Drafting Skill (Now Assist) generates a structured KB article from the incident cluster's resolution notes, work notes, and short descriptions.

Route → Review → Publish (Custom Flow)

Custom orchestration flow handles what OOB doesn't: route the draft to the correct product ownership team, manage the PO edit/re-draft cycle, escalate to KM for final review, and publish via the OOB article lifecycle.

Agentic Architecture

What makes this agentic? Traditional KB automation runs a fixed script. This workflow uses autonomous agents that perceive state (article status, work note content, CMDB data), make decisions (route, draft, revise, escalate), and act (invoke AI tools, assign records, send notifications) — with human approval gates at each critical decision. The platform detects. The orchestration agent decides. Humans approve.

ORCHESTRATION AGENT
WHY

Agentic Workflow

The business problem being solved: knowledge gaps accumulate silently. This workflow makes gap resolution autonomous — detect, draft, route, review, publish.

HOW

Orchestration Agent

A background reasoning engine that coordinates the workflow — dispatching agents, managing state, routing between review gates, and handling re-draft retries. It runs silently; the agents and workflow are the story.

WHO

Specialist Agents

Each agent owns a step: the Drafting Agent generates the article, the Revision Agent refines it with PO feedback, and the human review gates (PO, KM) approve or redirect.

WHAT

Agent Tools

Each specialist agent has its own tools — the capabilities it needs to perform its task. Tools are not shared; they belong to the agent that uses them.

Data Model

Zero custom schema. No custom fields on kb_knowledge. No custom tables. This build uses only OOB fields.

Traceability — Work Note (not a custom field)

When the flow creates the article, it immediately adds a work note:

"Article generated by Knowledge Gap Detection flow. Source incident cluster: INC0001234, INC0001235, INC0001236, INC0001237"

This gives the demo traceability talking point — "every AI-drafted article is traceable to the incidents that triggered it" — with no schema changes.

OOB Fields Used by the Flow

TableFieldHow We Use It
incident short_description, work_notes, close_notes, cmdb_ci Source material for Knowledge Demand Insights clustering and for the Knowledge Drafting Skill.
cmdb_ci assignment_group The routing key. incident.cmdb_ci.assignment_group determines which team receives the draft article for review. Set on the demo CI — no lookup table needed.
kb_knowledge workflow_state, assignment_group, kb_knowledge_base, work_notes workflow_state transitions drive the OOB lifecycle. assignment_group routes between PO team and KM team. work_notes carries traceability and rejection feedback.
sys_user_group name, sys_id Product ownership group (from cmdb_ci) for PO review, and KM Team group (Becki Love's team) for final review and fallback routing.

Routing Architecture

Zero custom logic. Routing uses a single OOB field already on the CMDB CI. No lookup table, no custom code, no import.

Routing Logic

// One line — read the assignment group directly from the CI target_group = incident.cmdb_ci.assignment_group // Fallback if CI is empty or has no assignment group IF target_group is empty: target_group = KM_TEAM_GROUP // Assign article and notify kb_article.assignment_group = target_group SEND notification N1 to target_group

For the POC Demo

We create the demo Webex CI with assignment_group set to the product ownership group. Since we control the demo data, routing is guaranteed correct with no additional logic.

Post-POC Path

If live CMDB data is inconsistent (CMDB assignment group = support team, not KB team), migrate to a reference table loaded from the SSO Product KB Mapping spreadsheet (11,163 rows, 1,676 products). No flow changes needed — just swap the lookup source.

Flow Designer Design

Main Flow: Knowledge Gap Orchestration

Trigger: KM team clicks [Generate Article] from the "Top 5 potential gaps" tab in Knowledge Center Insights (Knowledge Demand Insights). Custom flow hooks onto this OOB action.

Actions: Invoke OOB Knowledge Drafting Skill → create article (set custom fields) → routing lookup → assign + notify → call PO Review Subflow

Subflow: PO Review Cycle

Input: kb_knowledge.sys_id, gap incident cluster, ownership_group.sys_id

Logic: Wait for article state change → branch Approved / Rejected-Rework / Rejected-Not-Needed → on rework: clone Drafting Skill with feedback context → loop → 3-iteration loop guard

Subflow: KM Final Review

Input: kb_knowledge.sys_id (PO-approved article)
Logic: Reassign to KM Team group → notify → wait for KM decision → Approved → OOB publish (workflow state transition) | Rejected → reassign to original ownership group + work note with KM feedback → notify PO → re-enter PO Review Subflow

Re-Draft Loop — How It Works

// When PO rejects with "needs rework": 1. Flow reads last work note added by PO (rejection reason) 2. Invokes CLONED Knowledge Drafting Skill (via Now Assist Skill Kit): - Clone adds feedback-aware prompt: "You are revising a KB article. Original incident context: [short_desc, work_notes, close_notes] PO Feedback requiring changes: [rejection work note text] Revise the article to address this feedback." 3. Cloned skill returns revised draft 4. Article content updated on existing kb_knowledge record (same sys_id — no new record created) 5. article.workflow_state reset to "draft" (OOB transition) 6. article.assignment_group = original ownership group 7. Notification N2 sent: "Revision [N] ready for review" Loop guard: after 3 iterations without PO approval → notify KM Team: "Article stuck in review — manual intervention needed"

Phase 3 — AI Skill Setup
T008
Verify Knowledge Demand Insights + Knowledge Center Entry Point
Confirm the "Top 5 potential gaps" tab is visible in Knowledge Center Insights after the demo incident cluster is created. Verify the Predictive Intelligence scheduled job has run and surfaced the Webex gap. T009
Clone Knowledge Drafting Skill for Re-Draft Loop
Clone the OOB Knowledge Drafting Skill via Now Assist Skill Kit (NASK). Add feedback-aware prompt: passes original incident context + PO rejection work note as additional context. Name: "Knowledge Drafting Skill — Feedback Revision". This is the skill invoked in the re-draft loop, not the original.
Phase 4 — Orchestration Flow
T010
Build Main Orchestration Flow
Trigger: [Generate Article] from Knowledge Center gap suggestion. Actions: invoke OOB Knowledge Drafting Skill → create article in Draft state → add traceability work note (incident cluster INC numbers) → read incident.cmdb_ci.assignment_group → assign to ownership group → send N1 notification → call PO Review Subflow.
T011
Build PO Review Cycle Subflow
Wait for article state change. Branch: Approved → call KM Final Review Subflow. Rejected-rework → read PO work note → invoke cloned Drafting Skill (T009) → update article → reassign to PO → send N2 → loop. Rejected-not-needed → close. Loop guard at iteration 3 → send N5 to KM Team.
T012
Build KM Final Review Subflow
Reassign article to KM Team group → send N3 notification → wait for KM decision. Approved → OOB publish (workflow state transition to Published). Rejected → reassign to original ownership group + add KM work note → send N4 → re-enter PO Review Subflow.
Phase 5 — Notifications
T013
Create Notifications N1–N5
N1: PO team — AI draft ready. N2: PO team — revision ready (with iteration #). N3: KM team — PO-approved, ready for final review. N4: PO team — KM returned for revisions. N5: KM team — loop guard alert (article stuck). All link to article record and include relevant context.
Phase 6 — Demo Readiness
T014
End-to-End Flow Test
Trigger full flow from Knowledge Center gap suggestion. Validate: article created with correct custom fields, routed to correct ownership group, re-draft loop fires on PO rejection, KM routing fires on PO approval, article publishes on KM approval.
T015
Demo Readiness Check
Verify all 5 notifications send correctly. Confirm demo personas can access and action articles. Validate published article is searchable in KB and u_source_incident links back to demo incident. Confirm no empty screens in the demo clickthrough path.
